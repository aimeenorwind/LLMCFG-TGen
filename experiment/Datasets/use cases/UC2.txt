1---------- From: A case study for generating test cases from use cases------7 path
Use case#l:Search link by description
Description：A use case searches a set of links by their description and shows the results. 
MainSequence
	1 The visitor asks the system for searching links by description. 
	2 The system asks for the description. 
	3 The visitor introduces the description. 
	4 The system searches for links which match up with the description introduced by the visitor. 
	5 The system shows the found results. 
AlternativeSteps
	3.1 At any time, the visitor may cancel the search, then the use case ends. 
	4.1 If the visitor introduces an empty description, then the system searches for all the stored links and step 5 is repeated. 
ErrorSteps
	4.2 If the system finds any error performing the search, then an error message is shown and this use case ends. 
	4.3 If the result is empty, then the system shows a message and this use case ends.

Actor: Database (DB)

	2. Customer selects the type of account. 
	3. DB requests the withdrawal amount. 
	4. Customer inputs the amount. 
	5. Customer collects the withdrawn money. 
	6. DB generates and dispenses a receipt.  
	7. Customer collects the receipt.

	5b. DB provides the current account balance. 
	5c. DB exits the option.

8---Report Emergency Use case----
Use Case Name: Report Emergency 
Actors: FieldOfficer, Dispatcher  
Goal: The Field officer reports the emergency to the Dispatcher by activating the function on the terminal. 
Pre-Condition(s): The FieldOfficer is logged into FRIEND.  
Main Flow:
	M1. FieldOfficer activates the 'Report Emergency' function of FRIEND terminal 
	M2. FRIEND responds by presenting a form to the FieldOfficer. 
	M3. FieldOfficer clicks on open form button and describes possible responses to the emergency situation [S1][S2] 
	M4. FieldOfficer clicks on submit button 
	M5. The Dispatcher reviews the submitted information [S3][S4][S5] 
	M6. Dispatcher submits the final report and then FRIEND displays the acknowledgment and response to the FieldOfficer. 
	M7. Unable to make a connection [A1]  
Sub Flow:
	S1. FieldOfficer select the emergency level, type, location, and brief description of the situation. 
	S2. The FieldOfficer also describes possible responses to the emergency situation 
	S3. Creates an Incident in the database 
	S4. Dispatcher selects a response
	S5. Dispatcher submits the response to the FieldOfficer 
Alternate Flow: 
	A1. Extend use case Connection Down [E1] 
Exceptional Flow: 
	E1. Connection lost between the FieldOfficer and Dispatcher. Then, FRIEND responds the FieldOfficer and Dispatcher to try again 
Post Condition(s): The FieldOfficer has received an acknowledgment and the selected response from the Dispatcher.

9---Open Incident Use case----
Use Case Name: Open Incident 
Actors: Dispatcher  
Goal: The Dispatcher manages the incident on the basis of emergency reported by the FieldOfficer 
Pre-Condition(s): Dispatcher receives the information from the field officer.  
Main Flow:  
	M1. Dispatcher select a response sent by field officer [E1] 
	M2. If Information is present, Dispatcher Include Create Incident UC [A1] 
	M3. On the basis of priority incident (low or high), Dispatcher allocate the resources to the incident [A2][A3] 
	M4. Dispatcher Include Manage Incident UC [S1][S2][S3] 
	M5. Dispatcher close the incident Include Close Incident UC  
Sub Flows:  
	S1. Dispatcher receives reports and allocates resources. 
	S2. Dispatcher coordinates the resources. 
	S3. Dispatcher document the incident activity (about the allocation of resources and the action taken)  
Alternate Flows:  
	A1. If didn’t receive the information by the field officer, or in the database, go back to the entry condition, remain in the idle mode. 
	A2. For the low priority incident, the dispatcher allocates the resources of its own and the resource allocated is then committed to the specified incident. 
	A3. For the highest priority incident, the dispatcher notifies the chief for the allocation of resource to the specified incident.  
Exceptional Flows:  
	E1. Unable to select the response (restart the database or try again) 
Post Condition(s): Dispatcher opens an Incident.

Use Case ldentify Occupancy Status

Precondition
	The system has been initialized.
	
Basic Flow
	1. The SeatSensor SENDS capacitance TO the system.
	2. INCLUDE USE CASE Self Diagnosis.
	3. The system VALIDATES THAT the occupant class for airbag control is valid and the occupant class for seat belt reminder is valid.
	4. The system SENDS the occupant class for airbag control TO AirbagControllnit.
	5. The system SENDS the occupant class for seat belt reminder TO SeatBeltControlUnit.
	Postcondition: The occupant class for airbag control has been sent to AirbagControlUnit. The occupant class for seat belt reminder has been sent to SeatBeltControlUnit.
	
Bounded Alternative Flow
RFS 2-3
	1. IF voltage error is detected THEN
	2. The system resets classification filters.
	4. RESUME STEP 1.
	5. ENDIF
	Postcondition: Classification filters have been reset.
	
Specific Altenative Flow
RFS 3
	1. IF the occupant class for airbag control is not valid and the occupant class for seat belt reminder is not valid THEN
	2. The system SENDS the previous occupant class for airbag control TO AirbagControlUnit
	3. The system SENDS the previous occupant class for seat belt reminder TO SeatBeltControlUnit
	4. RESUME STEP 1.
	5. ENDIF
	Postcondition: The previous occupant classes for airbag control and seat belt reminder has been sent to AirbagControlUnit and to SeatBeltControlUnit respectively.
	
Specific Altenative Flow
RFS 3
	1. IF the occupant class for seat belt reminder is not valid THEN
	2. The system SENDS the occupant class for airbag control TO AirbagControlUnit.
	2. The system SENDS the previous occupant class for seat belt reminder TO SeatBeltControlUnit.
	4. RESUME STEP 1.
	5. ENDIF
	Postcondition: The occupant class for airbag control has been sent to AirbagControlUnit and the previous occupant class for seat belt reminder has been sent to SeatBeltControlUnit.

Use Case Self Diagnosis

Precondition
	The system has been initialized.
	
Basic Flow
	1. The system sets temperature errors to not detected.
	2. The system sets memory errors to not detected.
	3. The system VALIDATES THAT the NVM is accessible.
	4. The system REQUESTS the temperature FROM the SeatSensor.

	6. The system VALIDATES THAT the temperature is below 50 degrees.

	Postcondition: Error conditions have been examined.

	Postcondition: The system has detected a MemoryError.

Use Case Classify Occupancy Status

Precondition
	The system has been initialized.
	
Basic Flow
	1. The system sets the occupant class for airbag control to Init.
	2. The system sets the occupant class for seatbelt reminder to Init.
	3. IF the system VALIDATES THAT the capacitance is above 600.
	4. The system sets the occupant class for airbag control to Occupied.
	5. The system sets the occupant class for seatbelt reminder to Occupied.
	6. An adult has been detected on the seat.
	
Specific Alterative Flow
RFS 3
	1. IF capacitance is above 200 THEN
	2. The system sets the occupant class for airbag control to Empty.
	3. The system sets the occupant class for seatbelt reminder to Occupied.
	4. EXIT
	Postcondition: A child has been detected on the seat.
	
Specific Alterative Flow
RFS 3
	1. IF capacitance is below 200 THEN
	2. The system sets the occupant class for airbag control to Empty.
	3. The system sets the occupant class for seatbelt reminder to Empty.
	4. EXIT
	Postcondition: The seat has been recognized as being empty.
